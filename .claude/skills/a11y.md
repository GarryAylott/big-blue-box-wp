# Skill: Accessibility

Apply this skill to every piece of HTML, PHP template, SCSS, and JS produced for the Big Blue Box Podcast theme. The target standard is **WCAG 2.2 AA** — this is non-negotiable and is the UK legal baseline under the Equality Act 2010.

When reviewing or generating code, check against this skill before marking anything as complete.

---

## Baseline standard

**WCAG 2.2 AA** — every new component, template, or interaction must meet this.

Where AA is met with minimal extra effort, reach for AAA (e.g. contrast ratios, link underlines). Note when you are doing so.

---

## Structure and semantics

- Use the correct HTML element for the job. No `<div>` buttons, no `<span>` headings.
- Heading hierarchy must be logical and sequential (`h1` → `h2` → `h3`). Never skip a level for visual styling reasons — fix the style instead.
- One `<h1>` per page.
- Landmark regions must be present and correct: `<header>`, `<main>`, `<nav>`, `<aside>`, `<footer>`. Do not use `role="main"` on a `<div>` when `<main>` exists.
- Lists of links (navigation, tag groups, related content) must be wrapped in `<ul>`/`<ol>`.
- `<section>` and `<article>` elements must have an accessible name (via `aria-labelledby` pointing to a heading, or `aria-label` if no visible heading is present).

---

## Keyboard navigation

- Every interactive element must be reachable and operable by keyboard alone.
- Tab order must follow the visual reading order.
- Custom interactive components (dropdowns, modals, accordions, tabs) must implement the correct ARIA pattern from the [ARIA Authoring Practices Guide](https://www.w3.org/WAI/ARIA/apg/).
- Focus must never be trapped outside of an intentional modal/dialog pattern.
- Modals: when opened, focus moves to the modal; when closed, focus returns to the trigger element.
- Never use `tabindex` values greater than `0`. Use `0` or `-1` only.
- Never remove focus outlines without providing an equally visible replacement. The theme's focus styles must be defined in SCSS and must meet 3:1 contrast ratio against adjacent colours (WCAG 2.2 criterion 2.4.11).

---

## Colour and contrast

- **Normal text (< 18pt / < 14pt bold):** minimum 4.5:1 contrast ratio against background.
- **Large text (≥ 18pt / ≥ 14pt bold):** minimum 3:1.
- **UI components and focus indicators:** minimum 3:1 against adjacent colours.
- Never convey information by colour alone. Always pair with text, icon, or pattern.
- Check contrast against all theme colour token combinations — light and dark contexts both apply. Reference `src/scss/global/_colour-tokens.scss`.
- Do not hardcode colour values in components. Use the theme's CSS custom properties.

---

## Images and media

- Every `<img>` must have an `alt` attribute — always, no exceptions.
- Decorative images: `alt=""` (empty string, not omitted).
- Informative images: alt text describes the content and function, not the appearance ("Fourth Doctor in a striped scarf" not "old photo").
- Complex images (charts, diagrams): provide a longer description via `aria-describedby` or adjacent text.
- `<picture>` elements with multiple `<source>` tags: the `alt` lives on the `<img>` fallback only.
- Audio players must have visible controls and a transcript or caption where dialogue/content is relevant.
- Autoplay is not permitted for audio or video.

---

## Forms and inputs

- Every `<input>`, `<select>`, and `<textarea>` must have an associated `<label>` — linked via `for`/`id`, not proximity.
- Never use `placeholder` as a substitute for a label.
- Error messages must be:
  - Specific (say what's wrong and how to fix it)
  - Programmatically associated with the field (`aria-describedby`)
  - Not communicated by colour alone
- Required fields: use `required` attribute; supplement with visible indicator and `aria-required="true"` for complex widgets.
- Group related inputs (radio groups, checkboxes) in `<fieldset>` with a `<legend>`.

---

## ARIA

Use ARIA only when native HTML semantics are insufficient.

- Prefer native elements over ARIA roles. `<button>` over `role="button"`.
- Do not add ARIA roles to elements that already have correct implicit roles.
- `aria-hidden="true"` on decorative icons and SVGs — ensure the parent still has accessible text.
- `aria-live` regions for dynamic content updates (search results, filter changes, notifications). Use `polite` by default; `assertive` only for genuinely urgent alerts.
- `aria-expanded` on toggle triggers (accordions, dropdowns).
- `aria-current="page"` on active navigation links.

---

## Motion and animation

- Respect `prefers-reduced-motion`. Any scroll-based animation, transition, or parallax effect must be disabled or significantly reduced when this media query is active.
- In SCSS:
  ```scss
  @media (prefers-reduced-motion: reduce) {
    // disable or simplify animations
  }
  ```
- In JS: check `window.matchMedia('(prefers-reduced-motion: reduce)').matches` before initialising motion-heavy behaviour.
- Never use animation as the sole means of conveying information.

---

## Interactive components — quick reference

| Component | Required pattern |
|---|---|
| Dropdown navigation | `aria-expanded`, `aria-haspopup`, keyboard: Enter/Space to open, Escape to close, arrows to navigate |
| Modal/dialog | `role="dialog"`, `aria-modal="true"`, `aria-labelledby`, focus trap, Escape to close |
| Accordion | `aria-expanded`, `aria-controls` on button, `aria-labelledby` on panel |
| Tab interface | `role="tablist"`, `role="tab"`, `role="tabpanel"`, arrow key navigation |
| Tooltip | `role="tooltip"`, triggered on focus and hover, Escape to dismiss |
| Alert/notification | `role="alert"` or `aria-live="polite"` |

## Testing checklist (before marking complete)

- [ ] Keyboard-only navigation works end-to-end for the component
- [ ] Tab order is logical
- [ ] Focus indicator is visible and meets 3:1 contrast
- [ ] All images have appropriate alt text
- [ ] Heading hierarchy is correct
- [ ] Colour contrast passes at all text sizes
- [ ] No information conveyed by colour alone
- [ ] Reduced motion respected
- [ ] Checked with browser accessibility tree (Chrome DevTools / Firefox Accessibility panel)
- [ ] ARIA attributes are correct and not redundant

## Tools

- Browser DevTools accessibility panel (Chrome or Firefox) — check the accessibility tree
- [axe DevTools](https://www.deque.com/axe/) browser extension — automated audit
- [NVDA](https://www.nvaccess.org/) (free) or macOS VoiceOver for screen reader testing
- [Colour Contrast Analyser](https://www.tpgi.com/color-contrast-checker/) by TPGi
- [ARIA Authoring Practices Guide](https://www.w3.org/WAI/ARIA/apg/) — reference for component patterns
